# 專案架構設計反思與重構計畫 (Reflection & Pivot)

經過兩週的 Playtesting Agent 架構設計與實作，我們在此決定**終止目前的微觀節點架構開發，並轉向更符合真實多智能體系統 (MAS, Multi-Agent System) 的架構設計**。這意味著我們將重新設計並簡化系統架構，建立一個更精簡、具備高度適應力的系統。

---

## 為什麼終止當前架構？

我們發現目前的微觀節點設計存在以下兩大核心問題：

### 1. 微觀管理的狀態機 vs. 自主智能體
* **現狀問題**：目前的設計將遊戲測試的每個微小步驟（例如觀察、動作選擇、動作驗證、執行、結果驗證、Bug 偵測）都拆成全域 Graph 中的獨立 Node。
* **反思**：這導致系統過於複雜，本質上並不像一個 MAS，反而更像是一個**由 LLM 驅動的確定性腳本 (Deterministic Script)**、一個**超大型且被微觀管理的狀態機 (State Machine)**，LLM 在其中僅僅扮演了「邏輯控制閥」的角色。這失去了 Agent 系統應有的自主推理與決策彈性。
* **行業對比**：真實的大多數 MAS 設計通常是**層級式 (Hierarchical) 或是 Planner-Executor-Critic 架構**，將具體的執行邏輯封裝在 Agent 內部的 ReAct Loop 中，而非暴露在全域狀態圖上。

### 2. 與 Playtesting 動態環境的目標背道而馳
* **環境特性**：Playtesting 面臨的是一個**動態、複雜且不斷變化**的環境（隨時可能出現加載延遲、非預期的彈出畫面/廣告、隨機事件、操作困惑等）。
* **現狀問題**：為了在微觀管理的狀態機中處理這些動態變化，我們必須在 Graph 中設計大量的「條件分支邊 (Conditional Edges)」，並且為了滿足單一 Node 的局部需求而不斷在全域 State 中加入各種 Attribute。
* **反思**：這導致全域 State 極度混亂、冗餘，且**整體架構變得極度脆弱**。任何一個小環節的 UI 變動或延遲，都容易導致整個狀態機崩潰或陷入死循環。

---

## 下一步重構方向

我們將重新設計一個更符合 MAS 理念的精簡架構：

1. **粗粒度智能體 (Coarse-Grained Agents)**：
   * **Planner Agent (規劃者)**：負責高階測試案例的規劃與進度追蹤，不介入具體點擊細節。
   * **Executor Agent (執行者)**：接收高階目標，在節點內部運行 ReAct (Reasoning + Action) 迴圈，利用 Playwright 自主與遊戲互動，僅在任務完成或失敗時向外回報。
   * **Critic Agent (審查者)**：負責對執行軌跡、日誌與截圖進行客觀的 Bug 偵測與 Checkpoint 驗證。

2. **精簡全域狀態 (Lean State)**：
   * 全域 State 將只保留核心的測試計畫、執行紀錄與 Bug 清單，將臨時變數與微觀截圖限制在智能體內部。

3. **提升系統適應力 (High Adaptability)**：
   * 透過給予 Executor Agent 充分的工具調用與內部自主修正空間，來優雅應對網頁加載延遲、隨機彈窗等動態環境。
